核心判断02 / 06
Main Thesis

Subagent 的价值，不在更多 AI 说话，而在切开责任边界。

Attention / Authority / Accountability
01
为什么要拆开

Attention

把研究、实现、审查分离，避免一个主 Agent 同时背负所有上下文噪音。

Authority

不同子任务拥有不同工具边界与权限半径，降低误操作扩散。

02
最终看什么

Accountability

每个执行单元都要有清晰输入、输出和验收条件。

Boundary

真正的性能提升来自边界清晰，而不是简单堆 Agent 数量。

定义与触发03 / 06
Definition & Trigger

两种框架，先天关注点不同。

Execution Loop vs Studio Governance
CodeX
执行闭环
围绕当前任务拆分、执行、验证、汇报，服务于一次性目标完成。
Claude Code
工作室治理
更强调长期角色、Skills、Hooks、Worktrees 和项目边界组织。
Key Difference

一个优先回答“这次任务怎么做完”，另一个优先回答“这类任务以后怎么稳定做”。

所以二者不是简单强弱关系，而是偏不同任务半径与治理深度。

落地原则05 / 06
Before / After

先拆职责，再谈 Agent 数量。

Old Pattern vs Better Pattern
01
旧做法

Before

一个超大主 Agent 包办研究、编码、测试、文档，所有风险混在同一上下文里。

After

把研究、实现、审查、测试拆给不同执行单元，再由主 Agent 用验收门汇总。

Implementation Rule

先问有几个责任边界，再决定需要几个 Agent；顺序反了，架构通常会变形。
